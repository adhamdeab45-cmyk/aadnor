# تقرير فحص ADNOR V300

تاريخ المراجعة: 14 يوليو 2026

## فحوص آلية ناجحة

تم تشغيل:

```bash
npm run qa
```

والنتيجة: `QA PASSED`.

كما تم تشغيل `npm audit --audit-level=moderate` للمشروع و`npm audit --omit=dev --audit-level=moderate` للدوال بعد تثبيت الحزم، وكانت النتيجة `0 vulnerabilities` في الحالتين.

الفحص يغطي:

- وجود ملفات المستخدم والأدمن والوكيل والسيرفر والقواعد والتوثيق.
- تركيب جميع ملفات JavaScript بواسطة `node --check`.
- صلاحية جميع ملفات JSON.
- عدم وجود معرفات HTML مكررة.
- سلامة الروابط والملفات المحلية المشار إليها من HTML.
- مطابقة جميع أسماء Cloud Functions التي تستدعيها الواجهة مع الدوال المصدّرة.
- وجود 33 دالة Cloud Functions بدون أسماء مكررة.
- غياب سوق ADN وكوده وحقوله المحذوفة.
- عدم وجود كتابة مباشرة من الواجهة إلى مسارات الرصيد أو السحب أو شحن الوكلاء.
- تحقق قواعد قاعدة البيانات من Custom Claim ومن حالة الأدمن الحية.
- منع تعديل أو حذف إثبات الدفع بعد رفعه.
- بدء الإيداع والسحب بحالة موقوفة وعدم وجود طريقة دفع مفعلة افتراضياً.
- استبعاد ملفات الدوال والإعدادات الخاصة من Firebase Hosting.
- عدم وجود ملف Service Account داخل المشروع.
- وجود الشريط الذهبي الرسمي وربطه بالنتائج والسحوبات والوكلاء الفعليين فقط.
- منع إعادة استخدام إثبات الإيداع.
- تثبيت بيانات دورة السحب من أول تذكرة.
- حفظ حالة استلام مكافآت المهام ومنع تكرارها.

## فحوص إضافية تمت أثناء البناء

- تحميل ملف `functions/index.js` بنجاح بعد تثبيت الحزم.
- فحص واجهات `index.html` و`admin.html` و`agent.html` في بيئة DOM آلية ونجاح العناصر الأساسية.
- استخدام مولّد أرقام تشفيري للتذاكر والعجلة.
- تثبيت بيانات كل دورة سحب من أول تذكرة.
- منع جدولة نتيجتين لنفس الدورة.
- منع إضافة الجائزة مرتين عند النشر.
- قفل مالي لشراء التذاكر والعجلة والسحب وشحن الوكيل.
- تحقق الحساب النشط قبل المكافآت والدعوات والعمليات المالية.
- سحب الجلسات القديمة عند تغيير صلاحية أدمن أو إيقاف وكيل.
- تسجيل التغييرات المالية والإدارية المهمة في سجل التدقيق.

## ما لا يمكن إثباته داخل الملف وحده

لا يمكن تنفيذ الاختبارات التالية من دون نشر الدوال والقواعد والوصول إلى مشروع Firebase الحقيقي:

1. Google Login وPhone OTP على الدومين النهائي.
2. إرسال SMS حقيقي وحدود مزود الهاتف.
3. صلاحيات Custom Claims بعد منح الأدمن والوكيل.
4. عملية مالية حية بين حسابات Firebase فعلية.
5. جدولة Cloud Scheduler ونشر نتيجة في وقتها الحقيقي.
6. رفع إثبات إلى Storage من الدومين النهائي.
7. مزود دفع آلي أو Webhook؛ لا يوجد مزود أو مفاتيح مقدمة للمشروع.
8. قانونية النشاط في دولة معينة.

## خطة اختبار ما بعد النشر

1. شغّل فحص الجاهزية داخل `admin.html`.
2. أنشئ ثلاثة حسابات منفصلة: أدمن، وكيل، مستخدم.
3. استخدم أرصدة اختبارية صغيرة.
4. اختبر الشحن، الإيداع، السحب، التذكرة، العجلة والنتيجة.
5. اختبر الفشل أيضاً: رصيد ناقص، حساب مجمّد، وكيل موقوف، نتيجة مكررة.
6. راقب Realtime Database وFunctions Logs وسجل التدقيق بعد كل عملية.
7. لا تفعّل طرق الدفع للعامة قبل نجاح الدورة كاملة.

## الحكم الفني

المشروع مكتمل كمصدر إنتاج منظم وقابل للنشر، واجتاز فحوص التركيب والربط والأمان الساكنة. يبقى النشر والاختبار الحي على Firebase خطوة تشغيل إلزامية قبل استقبال مستخدمين أو أموال حقيقية.
