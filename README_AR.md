# ADNOR LEGENDARY V300 — مشروع الإنتاج الكامل

هذه النسخة إعادة بناء منظمة لـ ADNOR، وليست طبقة تصميم فوق الملف القديم. تم حذف سوق ADN نهائياً، وفصل الواجهات عن العمليات الحساسة، ونقل الرصيد والتذاكر والألعاب والسحب والوكلاء إلى Firebase Cloud Functions.

## ما يحتويه المشروع

- `index.html`: صفحة عامة قبل التسجيل + تطبيق المستخدم بخمسة أقسام.
- `admin.html`: لوحة إدارة مستقلة للكمبيوتر والتحكم التشغيلي.
- `agent.html`: بوابة وكيل الشحن مع الفحص والإيصال والسجل.
- `bootstrap.html`: تفعيل أول حساب أدمن بطريقة محمية بسر Firebase Secret.
- `assets/`: التصميم، JavaScript، اللغات، الأيقونات وصورة المشاركة.
- `functions/`: 33 دالة سيرفر للعمليات المالية والإدارية والجدولة.
- `database.rules.json`: قواعد Realtime Database مع تحقق صلاحية الأدمن الحية.
- `storage.rules`: حماية إثباتات الدفع ومرفقات الدعم ومنع تعديلها بعد الرفع.
- `database.seed.json`: إعدادات بداية آمنة، مهام، ألعاب وطرق دفع غير مفعلة.
- `scripts/`: فحص المشروع، الترحيل، تعيين الأدمن وإضافة بيانات البداية.
- `legal/`: الشروط والخصوصية مع إمكانية استبدال النص من لوحة الأدمن.
- `manifest.webmanifest` و`sw.js`: تثبيت الموقع كتطبيق وصفحة Offline.
- `DEPLOY_CHECKLIST_AR.md`: قائمة فحص الإطلاق.
- `SECURITY_AR.md`: تعليمات الأمان الأساسية.

## أهم مبدأ أمني

لا توجد دالة احتياطية تضيف أو تخصم رصيداً من المتصفح. إذا لم تكن Cloud Functions منشورة، ترفض العمليات المالية التنفيذ بدلاً من العمل بطريقة غير آمنة.

العمليات التالية تمر من السيرفر:

- شراء التذاكر وتثبيت سعر وجائزة وموعد كل دورة من أول تذكرة.
- تدوير العجلة بمولّد أرقام آمن وحدود وأوزان يحددها الأدمن.
- طلبات الإيداع والسحب والمراجعة وحجز مبلغ السحب.
- تعديل رصيد المستخدم ومنح وإلغاء VIP والتجميد.
- شحن الوكيل بخصم وإضافة ذرية واحدة وإيصال مرجعي.
- مكافأة الدخول والمهام والدعوات وحدود منع الإساءة.
- جدولة نتائج السحوبات ومنع تكرار نتيجة نفس الدورة ونشرها مرة واحدة.
- إنشاء الوكلاء وتفعيلهم وإيقافهم وسحب جلساتهم القديمة.
- منح وسحب صلاحية الأدمن مع تحقق حي داخل الدوال والقواعد.

## حالة البداية الآمنة

`database.seed.json` يبدأ بالشكل التالي عمداً:

- الإيداع **موقوف**.
- السحب **موقوف**.
- طرق الدفع موجودة كقوالب لكنها **غير مفعلة**.
- بريد الدعم فارغ حتى تدخل بريداً حقيقياً.

لا تفعّل الإيداع أو السحب قبل إدخال حسابات وتعليمات حقيقية واختبار دورة مالية كاملة.

## قبل النشر

1. خذ نسخة احتياطية من Realtime Database وStorage.
2. استخدم مشروع اختبار أو رابط Staging أولاً.
3. يلزم Firebase Blaze لأن المشروع يستخدم Cloud Functions والجدولة.
4. فعّل Authentication:
   - Google للمستخدم والأدمن.
   - Phone للمستخدم.
   - Email/Password لوكلاء الشحن.
5. فعّل Realtime Database وStorage.
6. أضف الدومين النهائي إلى Authorized domains.
7. راجع `DEPLOY_CHECKLIST_AR.md` و`SECURITY_AR.md`.
8. راجع قانونية السحوبات والألعاب واستقبال الأموال والعمر والهوية في الدول المستهدفة.

## تثبيت الأدوات وتشغيل الفحص

من داخل مجلد المشروع:

```bash
npm install
cd functions
npm install
cd ..
npm run qa
```

لا تنشر نسخة لا يمر فيها `npm run qa` بنجاح.

## إعداد Firebase والنشر

```bash
npm install -g firebase-tools
firebase login
firebase use adhamnnn-8a4d2
firebase functions:secrets:set ADMIN_BOOTSTRAP_KEY
```

اكتب كلمة سر طويلة لا تضعها داخل الملفات. بعد أخذ نسخة احتياطية، ادمج بيانات البداية:

```bash
firebase database:update / database.seed.json --project adhamnnn-8a4d2
```

ثم انشر:

```bash
firebase deploy --only database,storage,functions,hosting
```

## تفعيل أول أدمن

1. افتح `https://YOUR-DOMAIN/bootstrap.html`.
2. ادخل بحساب Google الرسمي.
3. اكتب نفس سر `ADMIN_BOOTSTRAP_KEY`.
4. اضغط تفعيل.
5. سجّل الخروج ثم ادخل مجدداً حتى يحصل الحساب على الصلاحية الجديدة.
6. افتح `admin.html` وشغّل فحص الجاهزية.

بديل متقدم باستخدام Application Default Credentials:

```bash
node scripts/set-admin.js admin@example.com
```

السكربت يمنح `admin`، يزيل صفة الوكيل إن وجدت، ويسحب الجلسات القديمة. لا تضع ملف Service Account داخل مجلد الاستضافة.

## ترحيل بيانات V200

السكربت لا يحذف البيانات القديمة. ينشئ الفهارس والملفات المفقودة، وينقل الإعدادات عندما لا توجد نسختها الجديدة، ويضيف بيانات الدليل العام للوكلاء مع عدد العمليات وآخر نشاط.

فحص فقط:

```bash
node scripts/migrate-v200.js
```

تطبيق بعد مراجعة القائمة:

```bash
node scripts/migrate-v200.js --apply
```


## الشريط الذهبي الرسمي

يبقى ظاهراً أعلى الموقع قبل الدخول وبعده، ولا يعرض أي رقم مصطنع. يتناوب فقط بين:

- آخر نتيجة نشرها الأدمن رسمياً.
- السحب القادم وجائزته وموعده.
- العدد الحقيقي للوكلاء الموثقين الظاهرين للعامة.

إذا لم توجد بيانات منشورة، يعرض حالة عامة بدون اختلاق فائز أو عدد مستخدمين.

## دورة السحب

1. يحدد الأدمن السعر والجائزة والموعد القادم.
2. عند بيع أول تذكرة ينشأ سجل دورة ثابت يحفظ السعر والجائزة والموعد.
3. تعديل إعدادات السحب القادم لا يغير حقوق تذاكر الدورة المفتوحة.
4. يختار الأدمن رقم تذكرة حقيقياً لنفس نوع ووقت الدورة.
5. النتيجة تبقى مخفية حتى موعدها.
6. `publishDueDraws` تعمل كل دقيقة، تضيف الجائزة مرة واحدة، تغلق الدورة وتنشر النتيجة.
7. يوجد فهرس يمنع جدولة نتيجتين لنفس الدورة.

## وكيل الشحن

- الأدمن ينشئ الوكيل من لوحة الوكلاء.
- اسم الدخول بلا `@` يتحول إلى بريد داخلي `@adnor-agent.com`.
- الوكيل يفحص المستخدم قبل التنفيذ.
- الخصم من الوكيل والإضافة للمستخدم عملية ذرية مع قفل مالي.
- كل عملية تحصل على إيصال ورقم مرجعي وسجل تدقيق.
- الرصيد الصفري يبقى صفراً، ولا يتم الرجوع إلى حقول قديمة.
- الدليل العام يعرض فقط الوكيل المفعّل والمسموح بإظهاره.

## ضوابط الطلبات والإثباتات

- يمنع السيرفر إعادة استخدام نفس إثبات الإيداع في أكثر من طلب.
- يحدد الأدمن أقصى عدد للودائع والسحوبات المعلقة وطلبات الدعم المفتوحة.
- لا تُقبل المكافآت أو العمليات المالية لحساب مجمّد.
- حالة استلام مكافأة المهمة لا تُمسح عند تكرار النشاط في اليوم نفسه.

## الدفع الآلي

المشروع يدعم الآن:

- إيداع يدوي بإثبات ومراجعة أدمن.
- سحب بمراجعة وحجز رصيد.
- شحن مباشر عن طريق وكيل موثق.

التكامل الآلي مع شركة دفع أو USDT Watcher يحتاج اسم المزود، حساباً فعلياً، مفاتيح API وWebhook خاصاً به. هذه البيانات لا يمكن اختراعها أو تضمينها بشكل آمن مسبقاً. عند اختيار المزود يجب إضافة Webhook داخل Cloud Functions والتحقق من توقيعه قبل إضافة الرصيد.
